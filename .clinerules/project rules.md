# 项目开发规则

## 1. 代码质量
- 使用 `npm run typecheck` 和 `npm run lint` 检查代码质量
- 代码提交前必须通过 typecheck 和 lint 检查
- 定期运行 `npm run check` 进行全面检查

## 2. 代码风格
- 遵循 TypeScript 严格模式，所有变量和函数必须有明确的类型定义
- 深入了解当前的 ESLint 规范要求，在开发过程中主动遵守，避免出现需要修复的问题
- 组件命名使用 PascalCase，文件命名与组件名一致
- 工具函数和 hook 使用 camelCase 命名
- 常量使用 UPPER_CASE 命名
- 使用路径别名 `@/` 代替相对路径导入
- 组件内使用 React Hooks，遵循 Hooks 规则

## 3. 测试规范
- 新功能必须添加相应的测试用例
- 修复 bug 后必须添加测试用例防止回归
- 运行 `npm test` 执行测试，确保测试覆盖率
- 定期运行 `npm run test:coverage`，目标覆盖率不低于 80%

## 5. 项目结构
- 组件目录按功能模块组织，包含组件文件、样式文件和测试文件
- 服务层统一放在 `src/services/` 目录下
- 工具函数放在 `src/utils/` 目录下
- 类型定义放在 `src/types/` 目录下
- 样式文件使用 Tailwind CSS，组件级样式使用 CSS Modules

## 6. 性能优化
- 组件使用 React.memo、useMemo、useCallback 等进行性能优化
- 避免不必要的重渲染，合理使用 React Hooks 依赖数组
- 图片资源使用懒加载，大型组件使用动态导入
- 状态管理保持精简，避免过度复杂化
- 使用 HTTP 缓存，实现请求合并和去重
- 优化 API 响应大小，减少第三方库的使用
- 避免在渲染过程中创建新函数或对象

## 7. 安全规范
- 敏感信息必须使用环境变量，不能硬编码
- API 请求必须使用安全的通信协议
- 用户输入必须进行验证，防止 XSS 和 SQL 注入攻击
- 定期更新依赖，修复安全漏洞
- 敏感数据必须加密存储和传输
- 不要在错误信息中泄露敏感信息

## 8. 文档规范
- 新功能必须更新相应的文档
- 代码中关键逻辑必须添加注释说明
- 组件和函数必须添加 TypeScript 类型注释
- 定期更新 README.md，保持文档与代码同步
- 为所有公共函数、组件和接口添加 JSDoc 注释

## 9. UI 设计规范
- 遵循设计系统规范，统一使用项目中定义的组件库
- 使用主题系统中定义的颜色变量，避免硬编码颜色值
- 确保颜色对比度符合 WCAG AA 级以上标准
- 使用统一的字体大小、行高和字重
- 确保在不同屏幕尺寸下都能正常显示
- 使用 Tailwind CSS 的响应式类：sm、md、lg、xl、2xl
- 使用 Lucide React 图标库
- 确保所有交互元素都有适当的 ARIA 属性，支持键盘导航

## 10. 数据库设计规范
- 项目使用远程数据库，所有数据库操作必须通过远程 API 或 Supabase 客户端进行
- 每个表必须有主键，推荐使用 UUID 类型
- 表名使用复数形式，使用 snake_case 命名
- 字段名使用 snake_case 命名
- 时间字段使用 timestamptz 类型，包含时区信息
- 为经常查询的字段添加索引，避免过多索引
- 所有数据库变更必须通过迁移脚本进行

## 11. API 设计规范
- 使用 RESTful API 设计，使用 HTTP 方法表示操作类型
- URL 路径使用 kebab-case 命名
- 使用适当的 HTTP 状态码和统一的错误响应格式
- API 端点命名必须清晰、语义化
- 使用 JSON 格式，响应数据必须包含明确的结构

## 12. 组件开发规范
- 组件必须具有良好的封装性和可复用性
- 组件 props 必须有明确的类型定义
- 大型组件必须拆分为更小的子组件，遵循单一职责原则
- 父子组件通信使用 props，跨组件通信优先使用 Context API
- 组件内部状态使用 useState，复杂状态逻辑使用 useReducer
- 组件必须有相应的测试用例，测试覆盖率目标 80% 以上

## 13. TypeScript 最佳实践
- 为所有变量、函数参数和返回值定义明确的类型
- 使用 interface 定义对象类型，type 定义联合类型和交叉类型
- 避免使用 any 类型，使用 unknown 代替并进行类型守卫
- 使用泛型增强代码复用性
- 使用 strict: true 模式

## 14. React Hooks 最佳实践
- 只在组件顶层调用 Hooks，不在循环、条件或嵌套函数中调用
- 为 useState 状态变量使用描述性名称
- 为 useEffect 提供明确的依赖数组
- 使用 useCallback 缓存回调函数，useMemo 缓存计算结果
- 自定义 Hooks 命名使用 use 开头

## 15. 服务层最佳实践
- 每个服务专注于一个业务领域
- 服务方法命名清晰，描述业务操作
- 服务层统一处理 API 请求和响应
- 服务层统一处理错误，返回标准化的错误信息

## 16. 项目特有规范
- 项目不使用用户系统及相关概念功能
- 所有设计和开发不考虑用户认证、授权、权限管理等相关功能
- 不存储和处理用户个人信息
- 所有数据处理必须考虑匿名性

## 17. 开发环境规范
- 开发环境、测试环境、生产环境必须分离
- 每个环境使用独立的配置文件
- 环境变量必须安全管理
- 使用统一的开发工具和插件
- 配置统一的编辑器设置，使用 Prettier 进行代码格式化

## 18. 持续集成与持续部署
- 实现自动化构建、测试、代码检查和部署
- 构建过程要包含类型检查和 lint 检查
- 部署前要运行测试，部署后要进行冒烟测试
- 实现监控与告警机制，定期分析监控数据

## 19. 可访问性
- 确保所有交互元素都有适当的 ARIA 属性
- 支持键盘导航
- 确保足够的颜色对比度（WCAG AA 级以上）
- 支持屏幕阅读器
- 提供替代文本：图片、图标等非文本元素必须有 alt 属性
